# ベストプラクティス — タイムラインの計画と管理

## Adobe Workfrontの「ベストプラクティス」とは

ベストプラクティスは、効果的で効率的な行動方針を示すガイドラインです。は、お客様やお客様の会社のユーザーに簡単に採用されます。とは、組織全体で正常にレプリケートできます。

これらの推奨事項を確認する際は、Workfrontのベストプラクティスの中には普遍的なものと、トピックに特化したものがあるものもあることに注意してください。 これらのベストプラクティスをフレームワークとして使用し、Workfrontのシステムの設定と使用をガイドします。

## タイムラインの計画と管理のベストプラクティス

* 完了したすべてのプロジェクトには、完了を反映したステータスが表示されます。

* プロジェクトをコピーする場合は、新しいプロジェクトのステータスを Planning に設定します。

* ユーザーがタスクに費やした実際の時間を記録して、実際の時間と予定時間を比較できるようにします。

* 特定の開始日と完了日を選択するのではなく、可能な場合は、タスク期間と先行タスクを使用して、プロジェクトタイムラインを作成および更新します。

* 各日（または設定されたスケジュールに従って）タスクのステータス、完了率、実際の時間を更新するようにユーザーに依頼します。

* プロジェクト計画を更新する際に、変更が行われたときに通知が自動的に送信されないように、プロジェクトのステータスを「計画」に設定します。

* プロジェクトで作業が割り当てられていないユーザーをプロジェクトチームから削除します。

* 主にWorkfrontを使用してデータを表示するユーザーは、左側のパネルメニューの上部に「プロジェクト指標」を配置します。

## これらのベストプラクティスはなぜですか？

**ベストプラクティス**

完了したすべてのプロジェクトには、完了を反映したステータスが表示されます。


**その理由は次のとおりです。**

完了するすべてのプロジェクトのステータスが「完了」（または同等の）になるようにすることで、Workfrontインスタンスが正常かつ最新の状態に保たれます。 プロジェクトのステータスを最新の状態に保ち、閉じることで、ユーザーは既に実行された作業を簡単に知ることができ、アクティブな優先度に焦点を当てることができます。 また、プロジェクト、タスク、リソースなどに関するレポートデータが正確になるようにします。


</br>
</br>

**ベストプラクティス**

プロジェクトをコピーする場合は、新しいプロジェクトのステータスを Planning に設定します。

**その理由は次のとおりです。**

計画ステータス（または同等のステータス）を使用すると、割り当てやタイムラインの変更などに関するWorkfrontの通知が、プロジェクトの準備が整う前に発生するのを防ぐことができます。 プロジェクトをコピーすると、プロジェクトオプションを含むダイアログボックスが表示されます。データが元のプロジェクトからコピーされたバージョンにコピーされないように、他のオプションを調整しながら、ステータスをここで変更します。

</br>
</br>

**ベストプラクティス**

ユーザーがタスクに費やした実際の時間を記録して、実際の時間と予定時間を比較できるようにします。


**その理由は次のとおりです。**

タスクの作業時間を把握することで、将来のプロジェクトを計画する際に正確性を高めるために、プロジェクトテンプレートを更新できます。 また、Workfrontのリソース管理ツールを使用した、より正確なリソース見積もりも使用されます。

</br>
</br>

**ベストプラクティス**

特定の開始日と完了日を選択するのではなく、可能な場合は、タスク期間と先行タスクを使用して、プロジェクトタイムラインを作成および更新します。

**その理由は次のとおりです。**

期間と先行タスクを柔軟なタスク制約（可能な限り早く、可能な限り遅く）と組み合わせて使用すると、プロジェクト計画を「カスケード」する自動タイムラインの日付変更が可能になります。 たとえば、タスクの期間が 1 日増えると、タスクの予定完了日が変更され、次のタスクの完了日が変更されます。

タスクの特定の開始日と完了日を選択すると、タスクの制約が日付を「ロック」する日付に変更されます（開始日、終了日、固定日）。つまり、タイムラインの日付を手動で更新する必要があります。

</br>
</br>


**ベストプラクティス**

各日（または設定されたスケジュールに従って）タスクのステータス、完了率、実際の時間を更新するようにユーザーに依頼します。

**その理由は次のとおりです。**

Workfrontのレポートは、Workfrontで入力されたデータと同じ精度です。 ステータスや完了率など、作業の進行状況を示す情報が定期的に更新されない場合、作業の進行状況を示すレポートは正確にはなりません。 毎日更新することで、リアルタイムレポートデータの精度を最大限に高めることができます。


タスクのステータスは、前の作業が完了し、新しいタスクを開始できることをユーザーに知らせるためにも使用します。 作業項目の実際の進捗状況を反映してタスクのステータスが変更されていない場合、Workfrontは適切な通知を送信できません。

</br>
</br>

**ベストプラクティス**

プロジェクト計画を更新する際に、変更が行われたときに通知が自動的に送信されないように、プロジェクトのステータスを「計画」に設定します。

**その理由は次のとおりです。**

プロジェクト計画の変更により、タスクの割り当て、計画開始日と完了日、およびその他の設定が変更された場合に、複数の通知を生成できます。 これは、ユーザーの混乱を招き、適切な割り当てや締め切りなどに関して混乱を招く可能性があります。

Planning ステータスは、プロジェクト計画がまだ開発中であるか、まだ実行する準備ができていないので、Workfrontに対してプロジェクトに関する通知をプロジェクトチームのメンバー（タスクやタスクを割り当てたユーザーなど）に送信しないよう伝えます。 変更が完了したら、プロジェクトのステータスを「現在」に戻すと、通知が送信されます。 このプロセスに従うと、ユーザーが受信する通知の量を最小限に抑えることができます。

</br>
</br>

**ベストプラクティス**

プロジェクトで作業が割り当てられていないユーザーをプロジェクトチームから削除します。


**その理由は次のとおりです。**

プロジェクトにタスクまたはイシューを割り当てると、プロジェクトの [ スケジュール設定 ] セクションと [ 担当者 ] セクションのプロジェクトチームリストにユーザーが追加されます。 ただし、割り当てから削除しても、プロジェクトチームリストに残ります。 これは、プロジェクトチームの一員として、プロジェクト上のアクティビティに関する通知を受け取り、「自分がいるプロジェクト」リストにプロジェクトが表示されるので、ユーザーが混乱する可能性があります。


また、プロジェクトチームメンバーは、プロジェクトとそのタスク、問題、ドキュメントに対する権限を取得します。 その結果、ユーザーがWorkfront内の不要な項目や不要な項目にアクセスできることがあります。

</br>
</br>

**ベストプラクティス**

主にWorkfrontを使用してデータを表示するユーザーは、左側のパネルメニューの上部に「プロジェクト指標」を配置します。

**その理由は次のとおりです。**

プロジェクトを管理したり、タスクの割り当てを実行したりしないリーダー、経営陣、その他のユーザーの多くは、プロジェクトを初めて開いたときに、このレベルのプロジェクト指標を見ることを望みます。 レイアウトテンプレートを使用して、プロジェクト指標をプロジェクトページの左のパネルメニューの一番上に移動し、ユーザーが見やすくアクセスしやすくします。
