# ベストプラクティス — プロジェクトテンプレート

## Adobe Workfrontの「ベストプラクティス」とは

ベストプラクティスは、効果的で効率的な行動方針を示すガイドラインです。は、お客様やお客様の会社のユーザーに簡単に採用されます。とは、組織全体で正常にレプリケートできます。

これらの推奨事項を確認する際は、Workfrontのベストプラクティスの中には普遍的なものと、トピックに特化したものがあるものもあることに注意してください。 これらのベストプラクティスをフレームワークとして使用し、Workfrontのシステムの設定と使用をガイドします。

* プロジェクトテンプレートを作成および更新できるユーザーのグループを設定します。

* テンプレートタスクに、タスクの期間、予定時間、および先行タスクが含まれていることを確認します。

* テンプレートの「スケジュール元」オプションを調整する場合は、タスクの制約を確認して更新します。

## これらのベストプラクティスはなぜですか？

**ベストプラクティス**

プロジェクトを作成する際には、テンプレートを使用します。

**その理由は次のとおりです。**

プロジェクトテンプレートは、プロジェクト管理者（およびプロジェクトを作成する他のユーザー）にとっての推測の対象となる、プロジェクトに含める必要のあるタスク、タイムラインの構造などを取り除きます。 テンプレートは、プロジェクトの作成を高速化する最も効果的な方法です。

重要な点は、テンプレートは、類似したタイプのプロジェクト間で一貫性を提供するので、ユーザー、プロセス、データポイントは、毎回同じ方法で詳細に記述できます。 すばやく切り替え（1 日または 2 日）ができ、最小限のタスクで済むプロジェクトでも、プロジェクトテンプレートを使用して作成すると便利です。

プロジェクト間でこの一貫性があると、より正確なデータが得られます。これは、チーム、部門、組織全体に対して意思決定をおこなう際に不可欠です。

</br>
</br>

**ベストプラクティス**

プロジェクトテンプレートの命名規則を設定します。

**その理由は次のとおりです。**

一貫した名前を付けると、テンプレートを見つけやすくなります。 また、複数のチームや部門に同じ名前のテンプレートが存在する場合、プロジェクトマネージャや他のプロジェクトを作成するユーザーは、適切なテンプレートを選択するのに役立ちます。

</br>
</br>

**ベストプラクティス**

プロジェクトテンプレートを作成および更新できるユーザーのグループを設定します。

**その理由は次のとおりです。**

作業管理と正確なレポートを行う上で、一貫した作成済みのプロジェクトテンプレートを持つことが重要です。 テンプレートを編集できるユーザー数を制限して、誤った変更や未承認の変更を防ぎます。

</br>
</br>

**ベストプラクティス**

プロジェクトテンプレートでプロジェクト共有を使用して、そのテンプレートを使用して作成されたプロジェクトへのアクセスを自動的に許可します。

**その理由は次のとおりです。**

特定のプロジェクトへのアクセスは、プロジェクト自体を通じて許可されます。 同じグループのユーザーが、特定のテンプレートで作成されたプロジェクトへのアクセスを常に必要とする場合は、テンプレートの「プロジェクト共有」オプションで追加します。 プロジェクトの作成後すぐにプロジェクトへのアクセスを制御できるだけでなく、将来権限を変更する必要がある場合のスケーラビリティの取り組みを合理化します。

**注意**:テンプレート共有は、テンプレート自体へのアクセスを許可します。 テンプレートを使用してプロジェクトを作成するには、少なくとも表示権限が必要です。

</br>
</br>

**ベストプラクティス**

個人ではなく、タスクにジョブの役割またはチームを割り当てます。

**その理由は次のとおりです。**

個々のユーザーが職階を変更したり、組織を離れたりした場合は、その人物を含むプロジェクトテンプレートを手動で更新する必要があります。 この作業は、システム管理者、グループ管理者、またはプロジェクト管理者の一部に時間がかかります。

テンプレートで職務の役割またはチームを使用する場合、その職務の役割またはチームを割り当てた人は誰でも作業を割り当てることができるので、スタッフの変更はプロジェクトテンプレートに直接影響を与えません。 これは、仕事が割れ目を通り抜けないようにするのに役立ちます。 また、Workfrontは、その役割に割り当てられた担当者のリストを表示できるので、職務の割り当てを個々のユーザーに容易に割り当てることができます。

また、Workfrontのリソース計画ツールでは、ジョブの役割を使用して、必要なリソースの計算や将来の作業の計画を立てることができます。

</br>
</br>

**ベストプラクティス**

プロジェクトテンプレートでタスクを作成する際は、精度が高すぎるのを避けます。 プロジェクトテンプレート内のタスク数を、作業の完了に必要なタスク数に制限します。

**その理由は次のとおりです。**

複雑すぎるプロジェクトテンプレートは、ユーザーにとって悪いエクスペリエンスとなります（プロジェクトマネージャー、リソースマネージャー、チームメンバーなど）。 タスクが多すぎると、タスクの期限が重なり、同じ役割または個人に複数のタスクが割り当てられるので、プロジェクトのタイムラインを管理しにくくなります。


</br>
</br>

**ベストプラクティス**

タスクの説明を使用して、複数のタスクに分割するのではなく、タスクの小さな手順を取得します。

**その理由は次のとおりです。**

1 つの行内の複数のタスクが同じジョブの役割または個人に割り当てられている場合は、それらのタスクを組み合わせることができます。 ユーザーに割り当てられたタスクが多すぎると、完了する作業が増えたように感じられ、Workfrontの導入に影響を与える可能性があります。

</br>
</br>

**ベストプラクティス**

テンプレートタスクに、タスクの期間、予定時間、および先行タスクが含まれていることを確認します。

**その理由は次のとおりです。**

期間、予定時間、先行タスクの 3 つは、プロジェクトのタイムラインの構成要素です。 これらは、作業にかかる時間と実行する必要があるタイミングを知るための鍵です。 Workfrontのリソース管理ツールの期間と予定時間、およびジョブロールの割り当てを使用して、リソース容量、稼動状況などを計算します。

期間や予定時間を初めて見積もる方法がわからない場合は、プロジェクトチームに依頼して初期見積もりを定義してください。 テンプレートを使用したら、プロジェクトチームに再度連絡し、テンプレートをより正確に変更できる場所を判断します。 ユーザーがWorkfrontで時間を記録している場合は、プロジェクトの予定時間と実際の時間を比較して、調整が必要な場所を確認できます。


</br>
</br>

**ベストプラクティス**

プロジェクトの詳細を事前設定し、カスタムフォームをテンプレートに添付します。

**その理由は次のとおりです。**

すべてのプロジェクトに標準の情報が、プロジェクトテンプレートに入力されていることを確認します。 これは、プロジェクトの作成を迅速にするのに役立つだけでなく、必要な情報がそこに存在し、プロジェクト間で一貫性が保たれるようにします。

テンプレートを使用して要求をプロジェクトに変換する際に、カスタムフォームを要求して送信された情報を取り込むための、プロジェクトのカスタムフォームを添付します。

</br>
</br>

**ベストプラクティス**

プロジェクトテンプレートを定期的に確認および更新します。

**その理由は次のとおりです。**

プロセスやチームが変更されたら、プロジェクトテンプレートを更新する必要があります。 定期的なケイデンス（四半期など）を設定して、どのテンプレートが積極的に使用されていないかを確認します。 これらは、まだWorkfrontにあるが、テンプレート選択リストには表示されないように、非アクティブ化することができます。

</br>
</br>

**ベストプラクティス**

テンプレートをチェックして、共有する前に必要な情報がすべて揃っていることを確認し、他のユーザーと共有して使用を開始します。


**その理由は次のとおりです。**

テンプレートはプロジェクトの作成に何度も使用されるので、すべてが正しく完全に設定されていることを確認する必要があります。 これにより、すべてのプロジェクトで一貫性が保たれ、レポートの正確なデータが得られます。

期間や予定時間などのタスク設定に加えて、テンプレートを共有する前に確認すべき点は次のとおりです。

* 設定からのスケジュール
* タスクの制約
* プロジェクトの所有者、スポンサー、グループ、会社
* Portfolio及びプログラム
* マイルストーンのパスと手順
* 承認プロセス
* プロジェクトでタスクを割り当てたユーザーが、プロジェクトに Contribute アクセス権を持っていることを確認します。
* リソース プール
* リマインダ通知
* スケジュール
* 通貨設定（該当する場合）
* 標準ドキュメントの添付
* 必要な税関フォームを添付します
* プロジェクトチームに余分な担当者が割り当てられていないことを確認します

</br>
</br>

**ベストプラクティス**

テンプレートの「スケジュール元」オプションを調整する場合は、タスクの制約を確認して更新します。

**その理由は次のとおりです。**

プロジェクトに異なるタスク制約を組み合わせると、予期しない混乱を招く予定日の計算がおこなわれる可能性があります。 たとえば、[ スケジュールの開始日 ] オプションで [ 開始日 ] を選択すると、既定では、そのプロジェクトで作成されたタスクに [ 可能な限り早く ] タスク制約が割り当てられます。 後で「スケジュールの開始日」オプションを「完了日」に切り替えた場合、作成したタスクにはデフォルトで「可能な限り遅延」タスク制約が設定されます。 意図しないタスクの組み合わせとこれらの制約が混在すると、プロジェクトタイムラインに計画日が混乱する可能性があります。


</br>
</br>

**ベストプラクティス**

テンプレートのプロジェクトチームを確認し、プロジェクトで作業しないユーザーを削除します。

**その理由は次のとおりです。**

既存のプロジェクトからテンプレートを作成すると、そのプロジェクトでタスクやタスクを割り当てられた担当者が引き継がれます。 また、テンプレートを使用しているときに、以前に作業を割り当てられた担当者を削除したり、テンプレートで自分で行った割り当てを変更したりできます。

これらのユーザーはすべて、プロジェクトチームの一部として、プロジェクトの「人」セクションと「スケジュール」セクションに表示されます。 その結果、テンプレートから作成されるすべてのプロジェクトに反映されます。 プロジェクトチームの一員として、プロジェクトのアクティビティに関する通知を受け取り、「自分が属するプロジェクト」リストのプロジェクトを参照し、プロジェクトとそのタスク、問題、ドキュメントに対する権限を取得するので、ユーザーは混乱を招きます。
