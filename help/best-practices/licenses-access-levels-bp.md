# ベストプラクティス — ライセンスとアクセスレベル

## Adobe Workfrontの「ベストプラクティス」とは

ベストプラクティスは、効果的で効率的な行動方針を示すガイドラインです。は、お客様やお客様の会社のユーザーに簡単に採用されます。とは、組織全体で正常にレプリケートできます。

これらの推奨事項を確認する際は、Workfrontのベストプラクティスの中には普遍的なものと、トピックに特化したものがあるものもあることに注意してください。 これらのベストプラクティスをフレームワークとして使用し、Workfrontのシステムの設定と使用をガイドします。

## ライセンスとアクセスレベルのベストプラクティス

* アクセスレベルを設定する際に、ユーザーのアクセスを減らして開始します。

* レビューおよびリクエストライセンスを割り当てる場合、通常はレビューにデフォルト設定されます。これは、Adobe Workfrontでユーザーに追加の権限を提供するからです。

* 特にユーザーが実行できるようにする場合を除き、すべてのアクセスレベルの各オブジェクトの「システム全体を共有」チェックボックスをオフにします。

* 「アクセスレベルで追加の制限を設定する」の下の「ユーザーによるコメントの削除を許可しない」設定を有効にすることを検討します。

* グループ管理者に優先するシステム管理者の数を制限します。

* 新しいアクセスレベルを一から作成するのではなく、既存のアクセスレベルをコピーして変更を加えます。

* 「説明」ボックスに、各アクセスレベルで実行できる操作を記録します。

* 作業目標の達成に必要なアクセスレベルのみに制限します。システム内のほとんどのユーザーのニーズを把握できるのが理想的なのは 4～5 人です。

* 少なくとも 2 人のユーザーにグローバルシステム管理者のアクセスレベルを割り当てます。

* アクセスレベルで機能を削除するのではなく、共有を通じてWorkfront項目でユーザーが実行できる操作を制限します。

## これらのベストプラクティスはなぜですか？

**ベストプラクティス**

アクセスレベルを設定する際に、ユーザーのアクセスを減らして開始します。



**その理由は次のとおりです。**

作業に必要な最小限のアクセス権でユーザーを起動します。 アクセス権が不十分なためにジョブを実行できない場合、通常は追加のアクセス権を要求します。 ユーザーにすぐに多くのアクセス権を付与すると、セキュリティ上の問題が発生する可能性があります。 さらに、アクセス権を取り消すよりも、常にユーザーにアクセス権を付与する方が効果的です。

</br>
</br>



**ベストプラクティス**

レビューおよびリクエストライセンスを割り当てる場合、通常はレビューにデフォルト設定されます。これは、Adobe Workfrontでユーザーに追加の権限を提供するからです。



**その理由は次のとおりです。**

レビューライセンスとリクエストライセンスの両方をWorkfrontのユーザー数に制限はありませんが、リクエストライセンスはリクエストをおこなって更新するだけのものに制限されます。 Review ライセンスは、Request ライセンスよりもプロジェクトやタスクにアクセスできる機能が多く、ポートフォリオやプログラムの表示、ドキュメントの編集、リソース管理ツールへのアクセスが可能です。

</br>
</br>

**ベストプラクティス**

ユーザーがシステム全体を共有できる必要がある特定の理由がない限り、すべてのアクセスレベルの各オブジェクトの「システム全体を共有」チェックボックスをオフにします。



**その理由は次のとおりです。**

システム全体でオブジェクトを共有することは、多くの場合、特定のユーザーがWorkfrontでアイテムを閲覧できるようにする松葉杖として使用されます。 これは、Workfrontグループの構造が不十分な場合や、共有権限が完全に理解されていない場合に発生します。 アイテムがシステム全体で共有されている場合、共有されているアイテムがすべてのユーザーに表示されます。 システムに保持されている情報の種類に応じて、プライバシーに関する問題が発生する可能性があります。



例えば、Workfront内の複数のベンダーと協力して、進行状況を確認したり、承認を提供したりできます。 「システム全体を共有」チェックボックスがオプションの場合は、そのチェックボックスをオンまたはデフォルトとして設定でき、すべてのベンダーが情報を利用できます。



完全にオプションをオフにすると、共有権限を持つユーザーが、会社、グループまたはチームを通じて、オブジェクトを共有する特定のユーザーまたはユーザーを決定する必要があります。

</br>
</br>

**ベストプラクティス**

「アクセスレベルで追加の制限を設定する」の下の「ユーザーによるコメントの削除を許可しない」設定を有効にすることを検討します。



**その理由は次のとおりです。**

このオプションを有効にすると、過去の通信がWorkfrontから削除されることを防ぐことができます。 監査目的で、コメント履歴全体を保持する必要がある組織もあります。

</br>
</br>

**ベストプラクティス**

グループ管理者に優先するシステム管理者の数を制限します。



**その理由は次のとおりです。**

システム管理者は、グローバルシステム設定を含め、Workfront内のすべての項目にアクセスできます。 設定グループ管理者がアクセスできるのは、システム管理者が制御し、その特定のグループにのみ適用します。



グループ管理者を使用すると、システム管理者は多くの責任を委任でき、Workfrontの日々のメンテナンスではなく、より大きな画像に焦点を当てることができます。 グループ管理者は、グループのニーズに容易に対応できるので、ユーザーに対するより優れたサービスを提供します。

</br>
</br>


**ベストプラクティス**

新しいアクセスレベルを一から作成するのではなく、既存のアクセスレベルをコピーして変更を加えます。



**その理由は次のとおりです。**

既存のアクセスレベルをコピーすると、新しいアクセスレベルに対して一貫性のあるベースが提供され、初期設定が同じになります。 また、システム管理者はアクセスレベルをゼロから完全に設定する必要がないので、これは時間の節約にもなります。

</br>
</br>

**ベストプラクティス**

「説明」ボックスに、各アクセスレベルで実行できる操作を記録します。



**その理由は次のとおりです。**

各オブジェクトタイプの設定の一覧を説明と共に説明します。 これにより、システム管理者（現在および将来）は、各アクセスレベルの動作を正確に把握できるので、設定を確認するためにアクセスレベル自体に進む必要はありません。



また、レポートで表示する際のアクセスレベルの比較が容易になります。 説明フィールドをビューにすばやく追加して、異なる違いや、異なるアクセスレベルが作成された理由をすばやく確認できます。

</br>
</br>

**ベストプラクティス**

作業目標の達成に必要なアクセスレベルのみに制限します。システム内のほとんどのユーザーのニーズを把握できるのが理想的なのは 4～5 人です。


**その理由は次のとおりです。**

アクセスレベルにより、Workfrontオブジェクトがユーザーと共有された場合、そのユーザーは編集や削除などに必要な権限を持つようになります。 個々の項目に対する共有をより具体的に設定できるので、アクセスレベルをより一般的に設定できます。


また、アクセスレベルが少ないと、システムの整備や戦略の実装が容易になり、会社や部署の切り替えに参加する際のオンボーディングが迅速になります。

</br>
</br>

**ベストプラクティス**

少なくとも 2 人のユーザーにグローバルシステム管理者のアクセスレベルを割り当てます。

**その理由は次のとおりです。**

複数のユーザーが、Workfrontがそのまま設定された理由、の管理/メンテナンス、ユーザーのサポート方法を理解する必要があります。 ある人が不在で、組織を離れたり、忙しいなどの場合は、情報と知識を持ち、システムを正常に管理できる別の人がいることを確認します。

</br>
</br>

**ベストプラクティス**

アクセスレベルで機能を削除するのではなく、共有を通じてWorkfront項目でユーザーが実行できる操作を制限します。


**その理由は次のとおりです。**

アクセスレベルは、グローバルレベルの特定の項目でユーザーが実行できる操作を制御します。 各プロジェクト、タスク、ポートフォリオ、ドキュメントなどに対する共有権限は、個々のユーザーがその特定のアイテムに対して実行できる操作を制御します。 特定のアクセスレベルを持つすべてのユーザーの機能を削除する代わりに、特定のアイテムに対する共有権限を微調整して、ユーザーが制限されるようにします。
